# MOZhi STAR Stories

> 用途：把每次项目变更沉淀成可直接复用到简历、面试、晋升材料里的 STAR 证据。
> 维护规则：新增一次较大的功能、重构、安全加固或稳定性优化时，继续往下追加，不要覆盖旧条目。
> 写法要求：每条都尽量落到真实文件、真实测试、真实指标或真实约束，不写空泛话术。

---

## 1. Phase 1 认证域生产化加固

### Situation

项目已经具备登录、注册、刷新会话、个人资料与头像上传能力，但认证入口仍保留开发态残留：挑战校验默认走本地 `noop`、文档仍指导使用 `dev-pass`、后端测试启动时还会冒出 Spring Security 默认用户警告，真实上线风险较高。

### Task

在不推翻现有 DDD 结构的前提下，把 Phase 1 的认证与用户域补齐到“可上线”的水位，同时保持前后端现有交互和页面改造不回退。

### Action

- 将挑战校验改为 Cloudflare Turnstile 服务端校验，并做 hostname allow-list 校验，核心实现落在后端 challenge verifier 与 gateway。
- 将 `cookie-secure` 和 `challenge.provider` 改成生产目标默认值，再通过 `dev/test` profile 显式覆盖本地行为。
- 通过排除 `UserDetailsServiceAutoConfiguration` 清理 Spring Security 默认用户残留，避免 JWT 服务与默认用户体系混淆。
- 用测试 provider 替换旧的 `dev-pass` 集成测试写法，保证测试不依赖外网，同时 challenge 升级链路仍然可回归。
- 前端把 challenge 文本输入升级为独立的 Turnstile widget 组件，避免认证页继续暴露伪交互。

### Result

- Phase 1 的核心 auth 链路仍可用，但默认配置不再指向开发态口子。
- 后端 challenge、用户域集成测试在 Java 21 下通过，且测试日志中不再出现 Spring Security generated password 警告。
- 前端认证页通过真实 widget 插槽承接挑战升级，后端与前端的安全语义保持一致。

### 面试可问

#### 你为什么不直接把 Turnstile 逻辑写在 Controller 里？

- 因为 challenge 是否需要、何时触发、触发后是否通过，属于 auth policy 的一部分；Controller 只负责接收 token，不应该承担 provider 细节。
- 我把“是否校验”留在 domain policy，把“怎么调 Turnstile”放在 infrastructure，这样之后替换 provider 不会污染业务域。

#### 你怎么保证测试环境不依赖外部验证码服务？

- 我单独加了 `test` provider，让测试环境走确定性 challenge verifier。
- 这样 challenge 升级链路依旧可测，但不会因为外部网络、密钥或第三方服务抖动而把 CI 变得不稳定。

#### 如果 Turnstile 配置错了会发生什么？

- 当前实现是 fail-closed。
- secret key 缺失、siteverify 请求异常、hostname 不在 allow-list，都会让 challenge 校验失败，而不是偷偷放行。

### 解决方法总结

- 安全策略归 domain，第三方 provider 细节归 infrastructure。
- 生产默认值收紧，开发例外通过 profile 打开。
- 所有可上线的安全行为都要有可回归测试，而不是只靠人工约定。

## 2. 认证前端从文本口令升级为真实 challenge 组件

### Situation

认证页虽然已经重做过 UI，但 challenge 升级仍停留在“出现一个文本框，手工输入 token”的过渡态。这种交互不仅不真实，还会误导使用者理解系统的真实能力边界。

### Task

把 challenge 交互改造成真实的验证码组件接入，同时保持登录/注册共用壳层、错误提示、刷新恢复和受保护路由逻辑不变。

### Action

- 将 Turnstile 逻辑抽出为独立的 `AuthChallengeWidget`，负责脚本加载、widget 渲染、token 更新、过期与重置。
- `AuthPage` 只保留“什么时候显示 challenge、什么时候重试提交”的页面状态，不再处理第三方脚本细节。
- 删除 `dev-pass` 辅助文案，统一改为“需要附加验证，请完成人机验证后重试”的真实语义。
- 补上认证页回归测试，约束 challenge 升级后必须出现 widget，而不是旧文本输入框。

### Result

- 认证页的视觉和交互语义终于一致，UI 不再假装存在一个并不真实的 challenge 流程。
- 组件边界更清晰，后面如果更换 provider 或补充交互样式，只需要改小组件而不是重写整页认证壳层。

### 面试可问

#### 为什么要单独拆一个 `AuthChallengeWidget`？

- 因为 `AuthPage` 已经承担登录、注册、文案、强度提示、跳转恢复等页面级职责，再把第三方脚本生命周期塞进去会让文件继续膨胀。
- 验证码这类集成最适合单独封装成“黑箱 UI 组件”，页面只拿结果 token。

#### 你怎么避免验证码 token 失效后用户还在重复提交旧 token？

- widget 有单独的 reset 信号。
- 认证失败后会清空本地 token 并重置 widget，防止 stale token 被重复复用。

### 解决方法总结

- 页面负责状态编排，组件负责第三方集成生命周期。
- 所有“看起来像真实能力”的 UI，都要尽快收敛到真实能力，而不是长期维持伪交互。

## 3. 个人中心复刻与工作台式信息架构重做

### Situation

原个人中心更偏功能表单页，和参考稿 `demo/personCenter.html` 的工作台式双栏结构差距较大，也承载不了“创作者中心”这类更强的产品气质。

### Task

把个人中心按参考稿重构为左侧身份卡 + 右侧工作台，同时保留现有真实后端能力，例如资料更新、头像上传、登录态与退出流程。

### Action

- 将个人中心重构为双栏布局：左侧资料卡、人群画像标签，右侧内容与创作中心 / 社交电商与生活两个工作台分组。
- 对没有后端接口的块使用明确的前端展示态，不伪造接口，不把展示态误写成真实业务能力。
- 将头像上传、昵称/简介修改保留在编辑态，继续调用现有用户域接口。
- 把工作台卡片图标全部替换为 SVG，避免纯色块或 emoji 式表达。
- 补上页面测试，约束关键标题、编辑视图切换、SVG 数量和文案边界。

### Result

- 个人中心从“表单页”提升成了“创作者工作台”。
- 参考稿的视觉语言被吸收进现有 React 应用，但真实接口边界没有被破坏。
- 页面层级和视觉识别更适合放进项目作品集或简历展示里。

### 面试可问

#### 为什么不把画像标签、AI 知识库这些块也一起做成后端接口？

- 因为这次任务目标是复刻与信息架构重做，不是凭空扩出新的后端子系统。
- 我保留了清晰边界：真实可写的资料字段继续走后端，未具备后端契约的块明确标成前端展示态，避免对产品能力做虚假陈述。

#### 为什么要把图标从色块换成 SVG？

- 因为这类工作台卡片本身就是用户扫视识别的核心锚点，纯色块的辨识度和完成度都不够。
- SVG 同时也更利于后续统一主题、hover、导出展示图。

### 解决方法总结

- 设计复刻不等于伪造后端。
- 复杂页面优先做信息架构重组，再补视觉细节。

## 4. 全局导航统一为“一级入口 + 搜索工具位”

### Situation

站点经过多轮页面扩展后，导航层级开始变散。搜索入口、一级栏目、登录态动作之间如果没有明确分工，后续再加页面会越来越难维护。

### Task

把导航统一成稳定的一级信息架构，并保证认证页、内容页、个人中心都共享一致的头部体验。

### Action

- 将全局导航稳定为 `首页 / 墨问 / 知选 / 实验室` 四个一级入口。
- 把搜索从“导航项”降为头部工具位，所有页面统一显示相同样式的搜索框。
- 让认证页也复用统一头部，而不是做成另一套独立导航。
- 补上登录态的 `退出登录` 动作，避免只有接口没有 UI。

### Result

- 导航职责更清楚：一级导航负责域入口，搜索框负责全站工具，认证动作负责身份态切换。
- 头部在多页面之间形成统一感，维护成本下降。

### 面试可问

#### 为什么保留四个一级导航而不是继续堆更多入口？

- 一级导航一旦过多，识别成本和后续扩展成本都会上升。
- 我保留最能代表站点主域的四个入口，把搜索放回工具位，避免信息架构继续膨胀。

### 解决方法总结

- 一级入口必须稀缺。
- 通用工具位不应挤占内容域入口。

## 5. 路由过渡与产品完成度提升

### Situation

在多页面应用里，如果切页是硬切，哪怕功能都能用，整体观感也会显得廉价，尤其是在作品集和面试演示场景里更明显。

### Task

在不引入复杂动画库和不打乱现有布局的前提下，为所有页面切换增加统一、可控、可降级的过渡感。

### Action

- 新增统一的 `RouteTransitionFrame`，只包裹路由内容区，不让导航和壳层跟着抖动。
- 使用轻量的 `opacity + translateY` 进入动画，时长控制在 220ms 左右。
- 兼容 `prefers-reduced-motion`，保证无障碍与低干扰。

### Result

- 页面切换更平滑，项目演示时的整体完成度明显更高。
- 动画只影响内容区，不会牵连导航布局，后续维护和扩展的风险较低。

### 面试可问

#### 为什么不用完整的页面出入场动画库？

- 因为这类项目里最值钱的是稳定和可控，而不是动画复杂度。
- 我选择的是壳层固定、内容轻过渡的方案，收益高、回归风险低。

### 解决方法总结

- 动画是产品完成度手段，不是目的。
- 能用统一外壳解决的问题，不要分散到每个页面各自实现。

## 6. 文档与资产治理

### Situation

随着页面迭代、设计比对和浏览器截图增加，根目录开始出现散落的 PNG 文件。如果继续放任，仓库会越来越像临时工作台，而不是可维护项目。

### Task

把截图、设计对比图、简历亮点沉淀、开发计划都放回结构化目录中，减少“只有作者自己知道”的隐性上下文。

### Action

- 新建统一的截图归档目录，收纳根目录散落的页面截图。
- 新建本文件，持续沉淀可复用的 STAR 面试故事。
- 将设计 spec、implementation plan、运维说明与页面截图分层保存，避免所有东西都堆在项目根目录。

### Result

- 项目从“能跑的仓库”更进一步变成了“可展示、可讲述、可维护的作品”。
- 后续无论是交接、写简历，还是准备面试，都可以直接复用仓库内的结构化材料。

### 面试可问

#### 为什么要专门维护项目亮点和资产归档？

- 因为真正影响工程效率的不只是代码，还有项目上下文是否能被复用。
- 我把这些材料沉淀进仓库，是为了让项目价值不只停留在“代码作者自己知道”，而是能稳定支持团队交接、简历包装和面试讲述。

### 解决方法总结

- 文档、截图、决策材料都属于工程资产。
- 能被下一次复用的材料，就不该散落在根目录或聊天记录里。
