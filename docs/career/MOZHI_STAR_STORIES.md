# MOZhi 项目 STAR 素材库

> 用途：把项目中的关键改造沉淀成可直接写进简历、项目介绍和面试回答的高质量素材。
> 维护方式：每次完成一个较大的功能、重构、安全加固或工程治理动作后，继续追加，不覆盖旧条目。
> 写法要求：少写“做了什么按钮和文案”，多写“为什么这样设计、怎么控制风险、如何验证结果、带来了什么工程价值”。

---

## 1. 主导认证域从开发态原型收口到生产默认安全基线

### 情境

项目在 Phase 1 阶段已经具备注册、登录、刷新会话、个人资料和头像上传等基础能力，但认证域仍残留明显的开发态实现：challenge 默认走本地绕过路径、Cookie 默认配置偏宽松、Spring Security 默认用户自动配置仍可能干扰真实鉴权体系。这样的系统可以演示，但不适合作为后续业务扩展的安全底座。

### 任务

在不推翻现有 DDD 六模块结构、不破坏前端已有交互的前提下，把认证域补齐到“可以继续往生产方向演进”的基线，并建立可持续回归的验证体系。

### 行动

- 将 challenge 校验从开发态本地通路收口为 Cloudflare Turnstile 服务端校验，并在后端增加 hostname 白名单校验，确保第三方能力接入后仍然保持服务端主导的安全边界。
- 将 `cookie-secure`、`challenge.provider` 等关键开关收紧为生产默认值，再通过 `dev` / `test` profile 显式覆盖本地与测试行为，避免“上线配置沿用开发默认值”的常见风险。
- 通过排除 `UserDetailsServiceAutoConfiguration` 清理 Spring Security 默认用户残留，确保系统只保留 JWT 双令牌模型这一套认证主路径。
- 为测试环境补充确定性的 challenge provider，切断外部验证码服务对 CI 的耦合，让认证升级链路既可测，又不会因为网络和第三方波动导致回归不稳定。
- 同步更新运行文档、环境变量契约和前端页面接入方式，保证安全方案不是“后端单边修改”，而是前后端、文档和测试一起收口。

### 结果

- 认证域不再依赖开发态绕过口子，默认行为更接近真实上线配置。
- 后端完整测试在 Java 21 环境下通过，前端认证页相关测试、构建和静态检查全部通过，安全升级具备完整回归闭环。
- 认证能力的结构边界更清晰：安全策略仍归业务域，第三方服务细节被限制在基础设施层，后续更换 provider 或继续补充风控手段时不会污染核心业务逻辑。

### 简历写法

- 主导认证域生产化加固，在不破坏既有 DDD 分层的前提下完成 challenge 服务端校验、Cookie 安全默认值收紧、Spring Security 默认链路清理和测试隔离，建立可持续回归的安全基线。
- 设计并落地“生产默认收紧、开发测试显式放宽”的配置治理策略，降低开发态配置误入上线环境的风险。

### 证据锚点

- 后端配置与启动收口：`mozhi-backend/mozhi-app/src/main/java/cn/zy/mozhi/app/config/AuthSecurityProperties.java`
- 启动层清理：`mozhi-backend/mozhi-app/src/main/java/cn/zy/mozhi/app/Application.java`
- 第三方校验接入：`mozhi-backend/mozhi-infrastructure/src/main/java/cn/zy/mozhi/infrastructure/adapter/port/TurnstileAuthChallengeVerifierPortImpl.java`
- 第三方网关实现：`mozhi-backend/mozhi-infrastructure/src/main/java/cn/zy/mozhi/infrastructure/gateway/turnstile/TurnstileSiteVerifyGateway.java`
- 回归验证：`mvnw -q -pl mozhi-app -am test`

### 面试追问与回答要点

#### 为什么不把第三方验证码逻辑直接写进 Controller？

回答要点：
把第三方校验逻辑直接写进 Controller，会让接入层同时承担协议解析、业务判断和第三方适配三种职责。我的处理方式是把“是否需要 challenge、校验失败后怎么处理”留在认证策略里，把“如何调用 Turnstile”放在 infrastructure，这样既符合 DDD 的依赖方向，也能在未来替换 provider 时把改动面控制在基础设施层。

#### 为什么强调“生产默认收紧，开发测试显式放宽”？

回答要点：
很多系统真正出问题，不是因为没有安全能力，而是因为开发态默认值一路带到了预发或生产。把默认值直接设成生产目标，再通过 profile 显式覆盖本地例外，能明显降低配置漂移风险，这是一种工程治理动作，不只是配置修改。

#### 为什么要额外做一个测试 provider？

回答要点：
安全链路必须能回归，但认证测试如果直接依赖第三方服务，会把 CI 稳定性和网络环境绑定在一起。测试 provider 的价值在于：既保留 challenge 升级路径的业务语义，又让测试具备确定性，这样验证体系才真正可靠。

## 2. 将第三方安全能力封装成可替换组件与标准接口

### 情境

很多项目接入验证码或风控组件时，容易把第三方脚本生命周期、页面状态和错误处理混在一起，最终造成前端页面臃肿、后端接口语义模糊、后续替换成本高。

### 任务

把第三方安全能力接入做成“页面只编排状态、组件只管理集成生命周期、后端只接收标准 token”的结构，而不是一次性把脚本和业务写死在页面或控制器里。

### 行动

- 将前端 challenge 能力抽象为独立的 `AuthChallengeWidget`，让脚本加载、组件渲染、token 更新、过期重置等逻辑集中在单点封装中。
- 让认证页只关心三件事：什么时候需要 challenge、什么时候允许提交、什么时候重置 token，从而保持页面层只负责状态编排。
- 后端保持 challenge token 只是认证请求的一部分输入，而不是暴露第三方 provider 细节给业务接口，维持契约稳定性。
- 在错误场景下增加 token 清理与 widget 重置逻辑，避免旧 token 被重复提交，减少因第三方状态残留导致的伪问题。

### 结果

- 第三方安全能力变成了“可替换、可回归、可边界化维护”的标准接入件，而不是耦合在页面里的临时脚本。
- 前端认证页职责更清晰，后端接口契约保持稳定，未来无论切 provider 还是扩展更多安全动作，改造成本都更低。

### 简历写法

- 设计第三方安全能力标准接入方案，将验证码接入拆分为页面状态编排、独立组件生命周期管理和服务端 token 校验三层，降低页面耦合与后续替换成本。
- 基于“最小暴露面”原则封装前后端 challenge 契约，避免第三方 provider 细节泄露到业务接口和页面主流程。

### 证据锚点

- 前端组件封装：`mozhi-web/src/components/auth/AuthChallengeWidget.tsx`
- 认证页面编排：`mozhi-web/src/pages/Auth/index.tsx`
- 接口契约与回归测试：`mozhi-web/src/pages/Auth/AuthPage.test.tsx`

### 面试追问与回答要点

#### 为什么要单独拆一个 challenge 组件，而不是直接在登录页写完？

回答要点：
页面负责登录、注册、错误提示、跳转恢复这些业务态已经足够复杂，再把第三方脚本生命周期塞进去会让页面继续膨胀。拆成组件后，页面只拿 token 和状态，第三方集成就被压缩成一个可以单测和替换的黑箱。

#### 如果以后从 Turnstile 换成别的验证码方案，这种封装能带来什么好处？

回答要点：
替换成本主要集中在组件内部和后端 gateway，不会扩散到整个认证页和业务域；这说明前面定义的边界是有效的。高质量封装的标准不是“今天能跑”，而是“明天替换时改动面可控”。

## 3. 将个人中心从功能表单页重构为工作台式创作者门户

### 情境

原个人中心更像一个资料维护表单，信息层次弱，无法承载创作者工作台、内容商业化和用户身份展示等更高一级的产品语义。如果直接继续在旧页面上堆功能，只会让结构越来越乱。

### 任务

在保持现有真实后端能力不回退的前提下，把个人中心重构成左侧身份卡、右侧工作台的双栏结构，并明确区分真实接口能力与前端展示态模块。

### 行动

- 以 `demo/personCenter.html` 为视觉与信息架构参考，将页面升级为“身份卡 + 工作台”的双栏布局，而不是继续在原表单页上增量堆叠。
- 对内容中心、社交电商、画像标签等暂时没有后端契约支持的模块明确采用前端展示态，避免通过伪接口或假数据把展示稿伪装成真实业务能力。
- 将昵称、简介、头像上传等已有真实能力继续保留在编辑态，并复用现有用户域接口，保证 UI 改造不会破坏真实业务闭环。
- 将工作台入口统一换成 SVG 图标和模块化卡片表达，强化扫视识别效率，避免“视觉看起来像占位稿”的低完成度问题。
- 为页面关键结构、编辑态切换和退出所有设备等动作补充测试，保证这次重构不仅仅是视觉改造，而是结构性重构。

### 结果

- 个人中心从“资料维护页”升级为“创作者工作台”，产品表达能力明显提升。
- 前端展示态与真实后端能力之间的边界被明确保留下来，后续业务迭代时不会因为这次 UI 重构而引入历史债务。
- 页面结构更适合作为作品集和简历项目展示，因为它展示的不只是界面，而是信息架构设计和能力边界治理。

### 简历写法

- 主导个人中心工作台化重构，将资料页升级为创作者门户，统一身份展示、内容入口和商业化模块布局，同时明确区分真实后端能力与前端展示态边界，避免视觉稿驱动下的伪业务实现。
- 在前端重构过程中坚持“真实接口不回退、展示模块不伪造后端能力”的治理原则，降低后续业务扩展的历史包袱。

### 证据锚点

- 参考稿：`demo/personCenter.html`
- 页面实现：`mozhi-web/src/pages/Profile/index.tsx`
- 页面回归：`mozhi-web/src/pages/Profile/ProfilePage.test.tsx`
- 样式与工作台图标：`mozhi-web/src/styles/globals.css`

### 面试追问与回答要点

#### 为什么不顺手把画像标签、知识库、拼团这些模块都补成真实接口？

回答要点：
因为这次目标是重构页面信息架构和能力边界，不是无计划地横向扩域。工程上我更重视“哪些是真的，哪些只是展示态”的边界清晰度，而不是为了让页面看起来完整就反向制造一堆没有设计过的数据契约。

#### 前端展示态和真实后端能力共存时，最重要的控制点是什么？

回答要点：
最重要的是不要让展示态污染真实契约，也不要让未来团队误以为这些模块已经具备后端支撑。我会在代码、文档和测试里同时体现这种边界，这样后续接手者不会被误导。

## 4. 统一站点壳层与跨页面核心能力，降低后续演进成本

### 情境

项目在持续扩展页面后，导航、搜索、登录态动作和页面切换体验逐渐分散。如果不尽早收口，认证页、内容页和个人中心会演化成多套头部和多套交互模型，后续每加一个页面都要重复维护。

### 任务

把导航、搜索、登出动作和页面过渡统一到稳定的站点壳层中，形成“页面只关心内容，壳层负责全局行为”的结构。

### 行动

- 将全局一级导航稳定为 `首页 / 墨问 / 知选 / 实验室` 四个入口，避免站点导航继续无约束膨胀。
- 把搜索从“栏目入口”重新定位为“全局工具位”，并在认证页、内容页和个人中心统一复用同一套头部样式与行为。
- 将退出登录、退出所有设备等登录态关键动作补齐到合理的全局位置，避免后端有能力、前端无入口。
- 新增统一的路由过渡壳层，只包裹内容区，不让导航和布局壳层一起抖动，从而在不引入复杂动画库的前提下提升整体完成度。

### 结果

- 站点壳层形成统一规范，后续新增页面只需要接入既有壳层能力，不必重新定义头部结构和全局动作。
- 导航职责更清晰，搜索、身份动作和内容入口各归其位，页面维护成本显著下降。
- 用户体验的一致性更好，项目整体完成度更适合用于作品展示和面试演示。

### 简历写法

- 统一站点壳层与跨页面公共能力，重构导航、搜索、登录态动作和页面过渡机制，形成“壳层负责全局行为、页面只承载业务内容”的前端架构模式。
- 通过统一头部与路由过渡外壳降低跨页面重复实现，提升多页面应用后续演进效率。

### 证据锚点

- 头部与导航：`mozhi-web/src/components/layout/AppHeader.tsx`
- 路由过渡：`mozhi-web/src/components/layout/RouteTransitionFrame.tsx`
- 壳层接入：`mozhi-web/src/layouts/AppShell.tsx`
- 认证壳层接入：`mozhi-web/src/layouts/AuthLayout.tsx`

### 面试追问与回答要点

#### 为什么要把搜索放在工具位，而不是继续作为一级导航？

回答要点：
一级导航承载的是站点主域入口，搜索本质上是跨域检索工具，不应该抢占内容域入口。把搜索从导航项中拿出来，本质上是在做信息架构治理，而不是单纯改样式。

#### 为什么路由动画只包内容区，不做整页切换特效？

回答要点：
因为整页动画虽然看起来更炫，但会把导航和布局壳层也卷进去，维护成本和回归风险更高。内容区轻过渡的方案收益更稳定，更适合真实工程项目。

## 5. 建立“规格先行、计划拆解、验证闭环”的交付方式

### 情境

复杂需求如果直接进入编码，往往会出现边界不清、实现漂移、测试补救式追加等问题。对于多模块项目，尤其是同时涉及前端、后端和文档的改动，更需要把需求执行过程结构化。

### 任务

把一次需求执行沉淀成“规格 -> 计划 -> 实现 -> 回归 -> 归档”的完整链路，让项目交付不仅仅依赖个人记忆，而是形成可复用的方法。

### 行动

- 先把关键改造点写成规格文档，明确安全收口边界、前后端职责和不做事项，避免实现过程跑偏。
- 再拆成可执行的 implementation plan，按测试先行和验证闭环推进，而不是一次性大改后再回头补测试。
- 在前端执行 `test / lint / build`，在后端执行完整 Maven 测试，并对提交前的 diff 做格式与空白检查，形成最小但完整的质量门。
- 把规格、计划、截图和 STAR 材料一并沉淀进仓库，而不是停留在聊天记录或临时说明中。

### 结果

- 需求从“口头对齐”变成“仓库内可追溯的闭环交付”，后续回看和接手成本更低。
- 这套流程天然适合 Agent 开发与多轮协作，因为上下文、验证标准和交付证据都被结构化保留下来。
- 对外讲项目时，不再只是讲“我做了什么功能”，而是能讲清“我如何组织一次跨前后端的高质量交付”。

### 简历写法

- 建立规格驱动的交付闭环，将需求执行沉淀为“规格定义、计划拆解、实现推进、质量验证、资产归档”五段式流程，提升跨模块改造的可追溯性与可复用性。
- 将前后端改造纳入统一验证门，避免只依赖人工自测或单侧测试判断交付完成。

### 证据锚点

- 规格文档：`docs/superpowers/specs/2026-04-08-phase1-production-hardening-design.md`
- 计划文档：`docs/superpowers/plans/2026-04-08-phase1-production-hardening.md`
- 页面重构计划：`docs/superpowers/plans/2026-04-08-person-center-replica.md`
- 前端验证：`npm run test`、`npm run lint`、`npm run build`
- 后端验证：`mvnw -q -pl mozhi-app -am test`

### 面试追问与回答要点

#### 这套流程和普通“先写代码再补文档”相比，核心收益是什么？

回答要点：
核心收益不是多了一份文档，而是减少返工。规格先行会提前暴露边界冲突和不做事项，计划拆解会让验证点前置，最后的回归和归档则保证这次修改不会只存在于作者脑子里。

#### 为什么这对 Agent 开发岗位有价值？

回答要点：
Agent 开发本质上很依赖上下文约束和验证闭环。如果没有结构化的规格、计划和证据，Agent 很容易把需求做散、做偏、做成一次性结果。把交付流程结构化，本质上是在为自动化和多轮协作提供稳定语境。

## 6. 推进工程资产治理，把仓库从“能跑”提升到“可维护、可展示、可复用”

### 情境

随着页面迭代、截图比对和多轮设计调整，仓库根目录容易出现散落图片、临时说明和只存在于聊天记录中的决策背景。这类问题短期看不影响运行，长期会直接拉高接手成本和作品包装成本。

### 任务

把截图、说明文档、规划文档和面试素材归档到稳定目录中，建立一套能长期复用的工程资产管理方式。

### 行动

- 新建截图归档目录与说明文件，把页面截图统一迁移到 `docs/assets/screenshots/` 下，避免根目录持续堆积临时图片。
- 新建 STAR 素材库，把高价值改造沉淀成可直接用于简历和面试讲述的结构化文本，而不是散落在聊天记录和提交信息里。

### 面试追问与回答要点

#### 为什么你会把截图、规划和面试素材也当成工程资产？

回答要点：
因为真实工程效率不只取决于代码，还取决于上下文能不能被复用。一个项目如果只有代码，没有结构化的背景、证据和材料，下一位接手的人就只能重新猜一遍。把这些内容纳入治理，是在降低长期协作成本。

#### 这类治理动作怎么体现高级工程师价值？

回答要点：
高级工程师不只是写功能的人，更是把系统做成“别人接得住、团队讲得清、后续改得动”的人。工程资产治理的本质，就是让项目价值可以持续复用，而不是随着个人上下线而蒸发。
