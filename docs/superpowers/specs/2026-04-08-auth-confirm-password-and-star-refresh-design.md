# 认证页确认密码与高阶 STAR 材料重写设计

## 背景

当前认证页注册态缺少“确认密码”输入，用户只能依赖单次输入完成注册，前端没有在提交前做最基础的一致性校验。与此同时，`docs/career/MOZHI_STAR_STORIES.md` 已经混入了较多页面与 UI 维度的素材，不符合“只保留高阶技术项目亮点”的目标。

## 目标

1. 在不修改后端注册接口契约的前提下，为前端注册表单补充“确认密码”输入与一致性校验。
2. 用户在两次密码不一致时，应在前端被阻断，不发起注册请求。
3. 重写 `docs/career/MOZHI_STAR_STORIES.md`，仅保留可用于高级后端 / Agent 开发岗位简历的高阶技术素材。

## 非目标

1. 不修改后端 `register` 接口 DTO 与领域服务签名。
2. 不引入邮箱验证、短信验证、找回密码等新认证流程。
3. 不保留本轮之前偏 UI 展示导向的 STAR 条目。

## 方案

### 认证页

- 注册态新增 `confirmPassword` 本地状态。
- 仅在 `mode === "register"` 时展示“确认密码”输入。
- 提交时先进行前端一致性校验：
  - 若两次密码不一致，直接展示明确错误文案并终止提交。
  - 若一致，则继续沿用现有 `registerAccount` 请求，只提交单个 `password` 字段。
- 切换登录 / 注册模式、注册成功后，需要重置确认密码状态与本地校验文案。

### STAR 材料

- 整体覆盖 `docs/career/MOZHI_STAR_STORIES.md`。
- 只保留高阶技术条目，聚焦以下主题：
  - 认证域生产化收口与安全默认值治理
  - 第三方 challenge 抽象与可替换接入边界
  - 本地 / 生产运行时分层、持久化与部署工程化
  - 面向 Agent 协作的规格驱动交付闭环
- 结构仍使用中文 `情境 / 任务 / 行动 / 结果`，但每条都要能直接映射到高阶岗位简历亮点。

## 测试策略

1. 为认证页新增失败测试：
   - 注册态应显示“确认密码”输入。
   - 两次密码不一致时，页面应提示错误且不调用 `registerAccount`。
2. 运行现有认证页测试，确保登录态与注册成功切换行为未回退。
3. 运行前端 `lint` 与 `build`，确认新增表单状态未引入类型和构建问题。

## 风险与控制

1. 风险：本地校验和服务端错误文案混杂，导致状态提示来源不清。
   控制：将“前端校验失败”作为独立提示来源，优先级高于接口返回错误。
2. 风险：STAR 文件重写后失去可追溯性。
   控制：新条目必须绑定真实代码文件、配置文件或验证命令，避免抽象空话。
