# 本地中间件持久化设计

## 目标

让本地 Docker 环境中的中间件在容器重建后仍保留状态，避免账号、对象存储资源、缓存落盘数据和 Kafka 日志在日常开发中被意外清空。

## 问题现状

- `docker-compose-environment.yml` 里的 MySQL、Redis、Kafka、MinIO 目前都没有数据卷
- 现在只有配置文件和初始化 SQL 被挂载，容器文件系统本身是易失的
- 一旦执行容器重建，本地账号和业务数据会直接丢失

## 方案

### 1. MySQL

- 使用命名卷挂载 `/var/lib/mysql`
- 保留现有初始化 SQL 挂载目录
- 首次建库仍由 `docker-entrypoint-initdb.d` 执行，之后数据由卷长期保存

### 2. Redis

- 使用命名卷挂载 `/data`
- 保持 `appendonly yes` 与 `appendfsync everysec`
- 这样 Redis 不只是“有卷”，而是会把 AOF 数据真正写进卷里

### 3. Kafka

- 使用命名卷挂载 `/var/lib/kafka/data`
- 保留当前单节点本地开发配置
- 目标不是生产级高可用，而是保证本地 topic / segment / offsets 不因容器重建而丢失

### 4. MinIO

- 使用命名卷挂载 `/data`
- 保证本地对象资源、bucket 元数据和上传产物在容器重建后仍可用

### 5. 契约测试与文档

- 扩展运行契约测试，要求环境 compose 为四类中间件都声明命名卷
- 文档明确说明：
  - `down` 不会删除数据卷
  - `down -v` 会删除数据卷并清空本地状态
  - 首次初始化与后续增量运行的差异

## 边界

- 这次只处理本地环境编排，不引入备份、恢复脚本或线上高可用方案
- 不会主动恢复已经丢失的历史账号数据；持久化从本次变更后开始生效
- 应用容器保持无状态，持久化范围仅限中间件数据目录

## 验证

- 契约测试必须覆盖 compose 中的中间件卷挂载
- `docker compose ... config` 必须能解析出命名卷
- 重建中间件容器后，命名卷仍然存在
- 文档必须说明持久化行为和清理方式
